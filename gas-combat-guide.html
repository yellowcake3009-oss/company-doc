What is GAS

GAS란 무엇인가?

Gameplay Ability System — 전투 시스템을 위한 언리얼 엔진의 공식 프레임워크

GAS(Gameplay Ability System)는 언리얼 엔진에 내장된 전투 및 게임플레이 시스템 프레임워크입니다.

캐릭터가 어떤 행동을 할 수 있는지(Ability), 그 행동의 결과가 무엇인지(Effect), 캐릭터의 능력치가 어떻게 변하는지(Attribute), 현재 어떤 상태인지(Tag)를 하나의 통합된 구조로 관리합니다.

원래 에픽게임즈가 포트나이트의 전투 시스템을 만들기 위해 개발했고, 이후 언리얼 엔진의 공식 플러그인으로 공개되었습니다. MMO, 액션 RPG, MOBA, 배틀로얄 등 전투가 있는 거의 모든 장르에서 활용할 수 있습니다.

Philosophy

GAS의 3가지 설계 철학

GAS는 왜 만들어졌고, 전투 기획자에게 왜 중요한가?

① 데이터와 로직의 분리

스킬의 "수치"(데미지, 쿨다운, 코스트)와 "실행 로직"(투사체 발사, 히트 판정)이 분리됩니다. 기획자가 수치를 바꿔도 프로그래머의 코드를 건드릴 필요가 없어요.

② 연출과 전투 로직의 독립

VFX, SFX, 카메라 셰이크 등 연출(Cue)은 전투 로직(Ability, Effect)과 완전히 분리됩니다. 연출팀이 이펙트를 바꿔도 밸런스에 영향이 없어요.

③ Tag 기반의 상태 관리

"스턴 중 스킬 불가", "슈퍼아머면 CC 무시" 같은 규칙을 코드가 아닌 Tag 설정으로 정의합니다. 기획자가 직접 전투 상호작용 규칙을 만들 수 있어요.

기획자에게 GAS가 중요한 이유: 기획서에 적는 내용이 시스템 구조와 1:1로 매핑됩니다.
"파이어볼은 MP 30 소모, 3초 쿨다운, 80 데미지 + 5초 화상" → 이게 그대로 Cost(Effect) → Cooldown(Effect) → Damage(Effect) → Burn DOT(Effect)으로 들어가요.
프로그래머와 동일한 언어로 커뮤니케이션할 수 있다는 것, 그게 GAS의 가장 큰 가치입니다.

기획자 포인트: 올드 MMO에서는 "스킬 시스템", "버프 시스템", "아이템 효과 시스템"을 각각 별도로 만들었습니다. 그래서 "스킬 버프와 아이템 버프가 중첩되나요?"라는 질문에 매번 프로그래머가 새 코드를 짜야 했어요.

GAS는 이 모든 것을 하나의 Effect 시스템으로 통합합니다. 스킬 버프든 아이템 버프든 음식 버프든, 전부 같은 Gameplay Effect이고 같은 Modifier 계산을 거쳐요. 덕분에 "어떤 출처의 버프든 동일한 규칙으로 중첩/우선순위가 처리된다"는 보장이 자동으로 됩니다.

Gameplay Tag

Tag(태그): GAS 모듈을 연결하는 "공용 라벨"

Gameplay Tag는 GAS 안에서 모든 모듈을 연결하는 계층형 라벨 시스템입니다. Ability의 발동 조건, Effect의 부여/면역, Cue의 트리거, Montage Notify의 이벤트 키 — 전부 Tag로 소통합니다. 코드 없이 전투 규칙을 정의하는 기획자의 핵심 도구예요. (다음 섹션에서 Tag가 GAS를 넘어 언리얼 전체에서 어떻게 쓰이는지 확장합니다.)

🏷 Tag는 "계층형 라벨"이다

Gameplay Effect

Effect(이펙트): "행동의 결과"를 정의하는 데이터

Gameplay Effect는 GAS에서 Attribute(숫자 스탯)를 변경하는 유일한 수단입니다. 데미지, 힐, 버프, 디버프, 쿨다운 — 전투에서 숫자가 바뀌는 일은 전부 Effect를 통해 일어나요. "얼마나 오래 지속되느냐"에 따라 3가지 타입으로 나뉩니다.

Effect가 하는 3가지 일: ① Modifier로 Attribute(숫자 스탯)를 변경하고, ② Granted Tag로 대상에게 상태 라벨을 부여하고, ③ Cue Tag를 발동시켜 연출을 트리거합니다. 이 세 가지가 하나의 Effect 에셋 안에 동시에 설정돼요. ①의 Modifier가 Attribute를 구체적으로 어떻게 계산하는지는 바로 다음 Pipeline 섹션에서 다룹니다.

Attribute

Attribute(어트리뷰트): Effect가 변경하는 "숫자"

HP, MP, ATK, DEF, SPD 같은 숫자 스탯이 Attribute입니다. 앞에서 배운 Effect가 바로 이 숫자를 변경하는 거예요. 모든 Attribute에는 두 개의 값이 있습니다: BaseValue(영구 기본값)와 CurrentValue(Modifier 반영값).

캐릭터 스탯 시트

Effect를 적용해보면 Base와 Current가 다르게 반응하는 것을 확인할 수 있습니다

⚠ GAS는 어떤 Attribute를 만들지 강제하지 않습니다. 아래의 MaxHP 분리, HP가 MaxHP를 초과하지 않도록 하는 상한 제한 등은 커뮤니티에서 권장하는 디자인 패턴이며, "Attribute 값이 변경되기 직전에 보정하는 규칙"을 프로그래머가 추가해야 합니다.

BaseValue vs CurrentValue — 핵심 차이:
· Instant Effect → BaseValue를 직접 변경 (영구 반영, 되돌릴 수 없음)
· Duration/Infinite Effect → CurrentValue만 임시 변경 (해제 시 Base로 복구)

위에서 "피격(HP -50)"은 Instant라서 Base 자체가 줄고, "ATK +30 버프"는 Duration이라 Current만 올라가고, "오라 MaxHP +100"은 Infinite라서 직접 해제 전까지 유지되는 것을 확인하세요. 그렇다면 여러 Effect가 동시에 하나의 Attribute를 바꾸면 어떻게 될까요? 다음 섹션에서 계산 순서를 다룹니다.

Effect × Attribute

계산 파이프라인: Effect가 Attribute를 바꾸는 과정

Effect와 Attribute를 각각 이해했으니, 이제 둘이 만나는 지점 — "여러 Effect가 하나의 Attribute에 동시 적용될 때 어떤 순서로 계산되는가"를 다룹니다. 전투 밸런싱의 핵심이에요.

아래 시나리오의 Effect들을 토글해보세요. 어떤 순서로 적용되고, 최종 ATK가 어떻게 나오는지 확인할 수 있습니다.

장비: 무기 ATK +40 (Infinite/Add)
스킬: 전투 함성 ATK +25 (Duration/Add)
음식: 고급 스테이크 ATK ×1.15 (Duration/Mul)
파티: 기사 오라 ATK ×1.2 (Infinite/Mul)

Effect를 토글하면 ATK 계산 과정이 단계별로 표시됩니다

⚠ 기획자 핵심 이슈: "총 공격력 50% 증가"는 어떻게 구현하는가?

"공격력 50% 증가" 버프가 2개 걸리면 어떻게 되어야 할까요? 이건 기획 의도에 따라 완전히 다른 결과가 나옵니다. 아래에서 두 방식을 비교해보세요.

방식 A: 곱연산 (Multiplicative)
각 버프가 순차적으로 곱해짐
GAS 기본 Multiply 동작
Base ATK = 100
버프A: ×1.5 → 100 × 1.5 = 150
버프B: ×1.5 → 150 × 1.5 = 225
실질 증가율: +125%
방식 B: 가산% (Additive Percentage)
%를 먼저 합산한 뒤 한번에 적용
커스텀 구현 필요 (커스텀 계산식 또는 중간 저장용 Attribute)
Base ATK = 100
버프A: +50%, 버프B: +50%
합산: 50 + 50 = 100%
100 × (1 + 1.0) = 200
실질 증가율: +100%
직접 비교해보세요 — 버프 수와 배율을 조절하면 두 방식의 차이가 드러납니다
구현 방식 정리:
곱연산GAS 기본 Multiply Modifier로 바로 가능. 간단하지만 버프 수가 많으면 기하급수적 증가. 가산%① "공격력 보너스%" 전용 Attribute를 만들어 Add로 %를 누적 → 커스텀 계산식에서 최종 값 산출
② 또는 중간 결과 저장용 Attribute를 두고 → 최종 Attribute에 반영 혼합"기본 스탯 증가"는 Add, "스킬 배율"은 곱연산, "장비 세트 보너스"는 가산% — 출처별로 다른 채널 사용
실전 사례: 원신(Genshin Impact)의 가산% 시스템
원신은 공격력 계산에 가산% 방식을 사용합니다.
최종 ATK = (기초 ATK + 무기 ATK) × (1 + 공격력% 합산) + 고정 공격력

기초 ATK — 캐릭터 레벨에 따른 고유 스탯 (예: Lv.90 후타오 = 106)
무기 ATK — 장착 무기의 기초 공격력 (예: 호마의 지팡이 = 608)
→ 기초 ATK + 무기 ATK = 106 + 608 = 714 (이것이 "백분율의 기준값")

예: 무기 부옵 +46.6%, 성유물 +46.6%, 불원소 공명 +25%
→ 공격력% 합산 = 46.6 + 46.6 + 25 = 118.2%
→ 714 × (1 + 1.182) = 1,558
→ 성유물 고정 공격력 +311 추가 → 최종 ATK = 1,869

만약 곱연산이었다면: 714 × 1.466 × 1.466 × 1.25 = 1,917 → 수치 차이는 작아 보이지만 버프가 쌓일수록 격차가 급증

GAS로 구현하려면: "공격력 보너스%" 전용 Attribute를 만들고, 각 버프가 Add Modifier로 %를 누적. 커스텀 계산식에서 (기초 ATK + 무기 ATK) × (1 + 보너스% 합산) + 고정 공격력 공식을 적용.

기획자 포인트: Add는 전부 합산 후 한번에 더하고, Multiply는 전부 곱합니다. 즉 "Add +25, Add +40"은 +65가 되고, "Mul ×1.15, Mul ×1.2, Mul ×0.7"은 ×0.966이 됩니다. 순서가 고정이라 "어떤 버프가 먼저 걸렸느냐"에 따라 결과가 달라지는 일이 없어요. 출처(스킬/아이템/파티 버프)와 무관하게 동일한 파이프라인을 거칩니다 — 이것이 GAS의 통합 설계가 주는 가장 큰 이점이에요.

Anim Notify란?

Montage 타임라인의 특정 프레임에 배치하는 이벤트 마커입니다. "이 프레임에 도달하면 → 이 이벤트를 발생시켜라"라는 의미예요. Ability는 WaitForEvent Task로 이 Notify를 기다리고 있다가, 신호가 오면 히트 판정이나 투사체 발사를 실행합니다.

기획자가 직접 하는 일

① 애니메이션에서 공격 모션의 "타점"을 찾습니다
② 그 프레임에 AN_SendGameplayEvent Notify를 배치합니다
③ Notify에 Event.Montage.Attack 같은 Gameplay Tag를 설정합니다
④ Ability가 이 Tag의 이벤트를 수신해서 히트 판정을 실행합니다
→ 타점 타이밍은 기획자가 Notify 위치로 조절!

Notify에서 VFX를 트리거하는 2가지 방식

① 직접 스폰 — 조건 없는 연출
Notify가 직접 파티클을 스폰합니다
GAS 로직을 거치지 않음 (Ability, Cue 불필요)
히트 여부와 무관하게 무조건 재생
예: 검기 잔상, 발자국 먼지, 무기 트레일
② GAS 경유 — 조건 있는 연출
Notify는 타이밍 신호만 Ability에 전달
Ability가 조건을 판단 → 결과에 따라 Cue 트리거
"히트 성공이면 타격 이펙트, 가드면 방어 이펙트"
예: 타격 이펙트, 크리티컬 연출, 버프 오라
기획자 포인트: "조건이 있으면 GAS를 거치고, 조건이 없으면 직접 스폰"이 판단 기준입니다.
검기 잔상은 빗나가도 보여야 하니까 직접 스폰, 타격 이펙트는 히트가 성공했을 때만 보여야 하니까 Ability → Cue 경로를 탑니다.
Gameplay Ability

Ability(어빌리티): 전투 컨텐츠 기획서의 "스킬 한 장"

지금까지 배운 Tag, Attribute, Effect, Cue, Montage — 이 모든 모듈을 하나로 엮는 것이 Gameplay Ability입니다. "기획서의 스킬 시트 한 장"에 대응하며, 발동 조건(Tag), 자원 소모(Effect), 애니메이션(Montage), 데미지(Effect→Attribute), 연출(Cue)을 조율합니다. 아래 에디터에서 파이어볼 스킬의 속성을 직접 수정하면, 지금까지 배운 모든 모듈이 어떻게 매핑되는지 확인할 수 있습니다.

Ability의 3단계: Grant → Activate → Execute

Ability는 "존재한다"와 "작동한다"가 분리되어 있습니다. MMO에서 스킬을 배우고(Grant), 스킬 키를 눌러 시전하고(Activate), 스킬이 데미지·연출을 수행(Execute)하는 것과 같은 구조입니다.

📦
① Grant (등록)
ASC에 Ability를 등록하는 단계.
"이 캐릭터는 이 스킬을 보유하고 있다."
· 캐릭터 생성 시 기본 부여
· 레벨업으로 스킬 습득
· 장비 장착 시 부여
· 스킬북 사용 시 부여
⚡
② Activate (발동)
Grant된 Ability를 시작시키는 단계.
TryActivateAbility() 호출 시점.
· Tag 조건 충족 여부 확인
· Block Tag 검사
· Cost(자원) 충분한지 체크
· 쿨다운 여부 확인
🔥
③ Execute (실행)
발동 성공 후 Ability 내부 로직을 수행하는 단계.
실제 게임에 영향을 주는 구간.
· Effect 적용 (데미지, 버프)
· Montage 재생 (모션)
· Cue 트리거 (이펙트, 사운드)
· Tag 부여 / 제거
예시 — "회피" Ability (GA_Dodge)

DETAIL

기획자 포인트: Ability 하나가 여러 Effect를 적용할 수 있습니다. 전사의 기본 공격조차 ① Cost Effect(스태미나 소모) ② Damage Effect(데미지) ③ Cooldown Effect(재사용 대기)를 동시에 실행합니다. 반면 회피는 데미지 Effect가 없고 무적 Effect만 있죠. 이처럼 같은 Ability 구조에 Effect 조합만 바꾸면 완전히 다른 스킬이 됩니다.

Montage가 있는 Ability vs 없는 Ability

Montage는 Ability의 선택 사항입니다. 모든 스킬에 모션이 필요한 건 아니에요. Montage 유무에 따라 Ability의 실행 흐름이 완전히 달라집니다.

⚔ 모션 스킬 (Montage 있음)
입력 → Tag 체크 → Cost → Montage 재생 → Notify 대기 → 히트 판정 → Effect → Cue
· 기본 공격, 콤보
· 시전형 마법 (캐스팅 모션)
· 회피, 구르기
· 피격 리액션 (HitReact도 Ability)
· 처형기, 잡기
⚡ 노모션 스킬 (Montage 없음)
입력 → Tag 체크 → Cost → 즉시 Effect 적용 → 끝
· 토글 버프 (오라 ON/OFF)
· 인스턴트 캐스트 (즉시 시전 스킬)
· 아이템/물약 사용
· 패시브 자동 발동 (HP 30% → 보호막)
· 장비 착용, 스탯 적용, 경험치 획득
기획자 관점: "이 스킬에 모션이 필요한가?"가 첫 번째 판단입니다. 모션이 있으면 Montage + Notify로 타이밍을 제어하고, 모션이 없으면 Effect만 즉시 적용. 어느 쪽이든 같은 Ability 구조(Cost, Cooldown, Tag 조건, Effect 적용)를 사용하기 때문에, 나중에 "즉시 시전이었던 스킬에 모션을 추가"하거나 반대로 바꾸는 것이 쉬워요.

Beyond GAS

Tag는 GAS만의 것이 아니다: 언리얼 전체의 공용 언어

┣ Attribute 숫자 스탯 관리 · HP, ATK, DEF 등 변화 감지 시 Tag 부여 가능 (HP 30% 이하 등) 직접 읽지 않음 (Effect의 Modifier가 변경) Effect의 대상
┗ Cue 연출 전담 · VFX, SFX, 카메라 직접 쓰지 않음 (소비자 전용) Cue Tag로 연출 에셋 매칭·실행 로직과 완전 분리
AnimBP 애니메이션 상태 관리 · 모션 재생·블렌딩 Notify로 Tag 이벤트 발생 (타점 등) Tag 감지 → 해당 모션으로 자동 전이 레이어별 독립
AI / BT NPC 의사결정 · "어떤 행동을 선택할 것인가" Task에서 GAS Ability 발동 → 간접 Tag 부여 Decorator에서 Tag 조건 분기 배타적 — 한 행동만
Enhanced Input 입력 매핑 · "어떤 입력을 받을 것인가" 직접 쓰지 않음 (소비자 전용) Tag로 IMC 활성/비활성 전환 (커스텀 로직) Priority 기반

기획자 포인트: Tag는 "게시판"이지, 어떤 시스템의 소유물이 아닙니다. StateTree가 보스를 Phase2로 전환하면서 Tag를 부여하면, GAS가 그 Tag를 읽어서 새 스킬을 활성화하고, AnimBP가 같은 Tag를 읽어서 분노 모션으로 바꾸고, Enhanced Input이 같은 Tag를 읽어서 회피 버튼을 긴급 회피로 바꿉니다. 기획자는 "어떤 Tag를 누가 쓰고 누가 읽는가"를 설계하면 됩니다. 이 설계가 곧 시스템 간 상호작용의 전부예요.
(참고: GAS 내부 모듈 간 Tag 통신은 기본 기능이지만, StateTree/AnimBP/Enhanced Input 등 외부 시스템과의 Tag 연동은 각각 커스텀 구현이 필요합니다.)

StateTree

StateTree: 캐릭터의 "Action State"를 관리한다

전투 캐릭터는 한 번에 하나의 행동만 할 수 있습니다 — 대기 중이면서 공격 중일 수는 없죠. StateTree는 이런 배타적 행동 상태(Action State)의 전환을 관리합니다. GAS Tag가 "동시에 여러 효과가 걸림"이라면, StateTree는 "지금 정확히 하나의 행동만 수행 중"이에요.

GAS Tag vs StateTree — 핵심 차이

GAS Tag (비배타 · 병렬)
"지금 걸려있는 효과/상태들"
스턴 + 화상 + 슈퍼아머가 동시에 공존
= 버프/디버프, 스킬 조건, 상태 이상
StateTree (배타적 · 직렬)
"지금 수행 중인 행동"
Idle / Attack / HitReact / Death 중 하나만
= Action State, 로코모션, AI 행동

🎮 캐릭터 Action State 시뮬레이터

전투 캐릭터의 Action State 전환을 시뮬레이션합니다. 버튼으로 이벤트를 발생시키면, StateTree가 상태를 전환하고 Tag를 부여하고, GAS와 AnimBP가 반응합니다.

⚠ StateTree ↔ GAS Tag 자동 연동은 기본 제공되지 않습니다. State 진입/퇴장 시 ASC에 Tag를 부여/제거하는 커스텀 StateTree Task를 만들어야 합니다. 아래는 그 결과를 시뮬레이션한 것입니다.

기획자 포인트: "공격 중에 피격" → StateTree가 Attack에서 HitReact로 전환 → State.Action.Attack Tag 제거, State.Action.HitReact Tag 부여 → GAS가 HitReact Tag를 읽어서 현재 공격 Ability를 Cancel → AnimBP가 같은 Tag를 읽어서 피격 모션 재생. 행동의 "전환"은 StateTree가, 전환의 "결과"는 GAS와 AnimBP가 Tag를 통해 처리합니다.
